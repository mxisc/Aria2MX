# AI 项目经验

- 症状：生产构建后的页面出现模板自带外部跳转入口，浏览器验证时会离开本地面板。原因：Vite 模板默认启用了 `vite-plugin-trae-solo-badge`。预防：面向实际发布的 Web 面板必须移除模板广告/徽标插件，并检查 `index.html` 标题与语言。
- 症状：本地冒烟测试访问 `127.0.0.1:18080` 命中了其他进程。原因：已有 IPv4 监听进程占用同端口，而测试服务监听在 IPv6 通配地址。预防：冒烟测试使用明确的 `127.0.0.1:<port>` 绑定，并用未占用端口。
- 症状：`.gitignore` 中写入 `ariamx` 会把源码目录 `cmd/ariamx/` 一并忽略。原因：无斜杠的 gitignore 规则会匹配任意层级同名文件或目录。预防：忽略根目录二进制时使用 `/ariamx`。
- 症状：冒烟验证中设置 `ARIAMX_ADMIN_PASSWORD=admin` 后仍无法登录。原因：`ARIAMX_ADMIN_PASSWORD` 只在配置文件首次创建时生效，已存在配置不会覆盖密码。预防：认证相关冒烟测试使用全新的 `ARIAMX_CONFIG` 路径。
- 症状：Vite 开发环境打开页面后 `/api` 请求会打到前端端口。原因：开发模式没有把 `/api` 转发给 Go 后端。预防：`vite.config.ts` 中配置 `/api` 代理到本地 Go 服务端口，并在 README 记录 dev 启动顺序。
- 症状：开发模式下前端能访问后端 API，但直接请求 `/mcp` 会返回前端 404。原因：Vite 代理只配置了 `/api` 和 `/jsonrpc`，没有把新增的 MCP 入口一起转发。预防：凡是前端开发态要直连的后端入口，都要同步写进 `vite.config.ts` 代理，MCP 场景至少包含 `/api`、`/jsonrpc` 和 `/mcp`。
- 症状：用户希望关闭 MCP 时不仅隐藏连接信息，还要让 `/mcp` 真正失效。原因：只改前端展示会留下后端入口仍可调用的假开关。预防：MCP 开关必须落到 `panel.mcpEnabled` 持久化配置，并同时控制设置页、连接信息页和后端 `handleMCP` 的实际访问行为。
- 症状：aria2 设置页只有裸 `key=value`，用户难以对应 AriaNg 的设置入口。原因：只读取了 `aria2.getGlobalOption`，没有维护 AriaNg 的分类和中文文案元数据。预防：全局选项页保留 AriaNg 的基本、HTTP/FTP/SFTP、HTTP、FTP/SFTP、BitTorrent、Metalink、RPC、高级分组，并为常用项提供中文名称、说明、只读和枚举信息。
- 症状：aria2 设置页新增高级 `key=value` 编辑区后 UI 被撑乱。原因：raw textarea 常驻在设置页下方，且 `.options-panel textarea` 全局选择器会影响该面板内所有 textarea。预防：高级编辑区默认折叠，并用 `.raw-option-editor textarea` 精确限定大文本框样式。
- 症状：aria2 设置页右侧标题在两栏结构外，右侧内容上下边界无法和左侧分类栏对齐。原因：标题和操作按钮不在 `.settings-layout` 同一网格内。预防：设置页标题、搜索和操作按钮都放进右侧 `.settings-content`，两栏容器只负责左右对齐。
- 症状：用户要求界面“像 AriaNg”，但局部组件调整后整体观感仍停留在玻璃拟态。原因：主题色、边框、按钮、列表和侧栏基线样式都集中在 `src/style.css`，只改单个组件无法完成整体迁移。预防：涉及整站风格切换时，优先统一全局设计 token 与共享容器/控件样式，再处理页面细节。
- 症状：用户提供了设计导出目录后，若只参考单个 HTML 页面，容易漏掉统一的颜色、字号和间距规范。原因：结构示例、设计 token 和说明分别散落在 `ariang-config-design.html`、`ariang-admin-style.css`、`ariang-admin-style-guide.md`。预防：遇到该目录下的样式对齐任务时，优先以 CSS 和 style guide 作为全局视觉基线，再用 HTML 对齐局部结构。
- 症状：用户要求“经典皮肤 / 设计稿皮肤”按账号保留时，只靠 `localStorage` 会在清缓存或换设备后丢失。原因：皮肤状态属于面板配置，不是纯浏览器临时偏好。预防：皮肤字段写入 `panel.theme` 并通过 `/api/config` 下发；前端启动和登录成功后都要重新应用服务端主题。
- 症状：all-in-one 模式启动内置 aria2 时，若运行时目录用相对路径或未预创建 `session.dat`，`aria2c` 会直接启动失败。原因：进程执行路径依赖当前工作目录，且 `--input-file` 指向的 session 文件必须存在。预防：内置 aria2 的状态目录统一转成绝对路径，并在首次启动前创建空的 `session.dat`。
- 症状：all-in-one 发布包体积异常膨胀到约 93M。原因：`//go:embed runtime/*` 会把 `darwin-arm64`、`linux-amd64`、`linux-arm64` 三套 aria2 运行时归档全部打进每一个二进制。预防：all-in-one 构建必须使用按 `GOOS/GOARCH` 拆分的 build tag embed，并让 CI 按平台分别产出发布包。
- 症状：面板里修改 `rpc-listen-port` 后界面显示已保存，但内置 aria2 实际仍监听旧端口。原因：Aria2 选项里的 `rpc-listen-port` 与面板托管字段 `ManagedRPCPort` 是两套状态，保存时没有同步。预防：all-in-one 模式下凡是由面板启动参数接管的 aria2 选项，保存时必须先映射回托管配置，再决定是否重启 aria2。
- 症状：修改内置 aria2 的 RPC 监听端口后，旧 aria2 先被停掉，但新 aria2 因端口已占用启动失败，结果面板还在、aria2 却掉线。原因：重启前没有先探测新端口是否可绑定，本机残留进程或其他服务占用端口时会把切换过程做成半失败。预防：保存 `rpc-listen-port` 前必须先探测 `127.0.0.1`/`::1` 端口可用性；若不可用，直接拒绝保存并保留当前 aria2 进程。
- 症状：移除旧皮肤枚举后，已存在配置仍保存 `design`/`classic`，前端和后端会出现主题状态不一致。原因：主题字段属于持久化配置，删除旧样式时如果不做迁移，历史配置不会自动收敛到新枚举。预防：主题重构时保留 `panel.theme` 兼容迁移逻辑，并把显示模式拆到独立的 `panel.colorMode` 字段；`design -> light`，`classic -> dark`。
- 症状：面板异常退出后，残留的内置 aria2 仍占着原 RPC 端口，下一次启动面板会误判成“端口冲突”而无法启动。原因：之前启动逻辑只检查端口是否可绑定，没有校验占用者是否就是上一轮遗留的同 secret aria2 进程。预防：all-in-one 启动时若发现 RPC 端口已占用，先用当前 `rpcUrl + rpcSecret` 调 `aria2.getVersion`；如果鉴权成功，就直接复用旧 aria2，而不是重复拉起。
- 症状：AIO 场景里，内置 RPC 固定端口被其他进程占用时，面板无法继续托管 aria2。原因：单一固定端口在用户本机容易与已有服务冲突。预防：当目标端口不可用且也不是可复用的旧 aria2 时，从目标端口开始按 `+10` 自动步进寻找空闲端口，并把最终端口写回配置；同时在主进程收到 `SIGINT`/`SIGTERM` 时主动停止内置 aria2，减少残留进程。
- 症状：主题模式加入“跟随系统”后，如果只把 `system` 原样写进 DOM 的 `data-mode`，现有深浅色 CSS 不会生效，也不会随系统切换自动更新。原因：样式层实际只识别解析后的 `light`/`dark`。预防：配置层允许保存 `panel.colorMode=system`，但前端应用主题时必须先解析成实际模式写入 DOM，并监听 `prefers-color-scheme` 变化实时重算。
- 症状：本地明明已经改了代码，但浏览器里看到的还是旧界面。原因：旧的 `dist/ariamx` 二进制实例仍在运行，而不是当前工作区代码启动的开发态服务。预防：当用户要求切到最新版本或实时查看改动时，先停掉旧的发布二进制实例，再用当前工作区代码启动后端，并确认前端访问的是对应的开发服务。
- 症状：前端源码里已经有新选项，但通过 Go 服务访问页面时仍看不到。原因：Go 端提供的是编译时嵌入的 `internal/web/dist`，仅重启 `go run ./cmd/ariamx` 不会自动把 `src/` 变更编进页面。预防：只要修改了前端可见行为且用户通过后端端口访问页面，就先执行 `pnpm run build` 更新 `internal/web/dist`，再重启 Go 服务。
- 症状：外部程序接入 `/jsonrpc` 时，如果直接复用 aria2 内部 secret，会把面板代理层和内置 aria2 的信任边界混在一起。原因：面板作为 RPC 中间件需要有独立鉴权，而 aria2 内部 secret 只应存在于面板到 aria2 的内部链路。预防：外部调用统一校验 `panel.rpcSecret`，并保证它永远不与 `aria2.rpcSecret` 相同；AIO 模式下内置 aria2 继续只监听 `127.0.0.1`。
- 症状：连接信息页需要展示面板 RPC Secret 时，若直接开放到匿名接口会扩大敏感信息暴露面。原因：secret 属于高敏感配置，只适合在受控页面里最小范围展示。预防：默认不要在匿名接口泄露 secret；只有在用户明确要求时，才通过已登录接口返回并显示在受保护页面中。
- 症状：改成面板独立 RPC Secret 后，AriaNg 默认的 `token:<secret>` 参数鉴权会失效。原因：AriaNg 不会主动补 `Authorization: Bearer`，而是把 secret 放进 JSON-RPC `params[0]`。预防：面板 `/jsonrpc` 需要兼容校验外部传入的 `token:<panel.rpcSecret>`，但校验通过后仍要丢弃它，并替换成内部 `aria2.rpcSecret` 转发，不能把外部 token 原样透传给 aria2。
- 症状：给面板增加 MCP 能力后，如果另起一套 secret 或匿名开放工具列表，外部客户端配置和信任边界会变复杂。原因：MCP 和 `/jsonrpc` 本质上都是面板代理层入口。预防：`/mcp` 继续复用 `panel.rpcSecret`，并在登录页内直接展示可复制的 MCP 地址和客户端配置示例。
- 症状：MCP 只实现 `tools/list` / `tools/call` 时，客户端虽然能调 aria2 工具，但无法按标准发现资源、Prompt，也不能对任务 GID 或资源模板参数做补全。原因：最小版 MCP 只覆盖了 tools primitive，没有把 `resources`、`prompts` 和 `completion` 一起实现。预防：AriaMX 的 MCP 服务保持在同一协议版本下完整声明并实现 `tools + resources + prompts + completion`，资源 URI、Prompt 名称和补全参数要和面板现有功能保持一致。
- 症状：失败任务点击“重新开始”后直接从列表消失，用户误以为面板把任务吞掉了。原因：按钮实际绑定的是 `aria2.removeDownloadResult`，删除了历史结果而不是重新创建下载。预防：失败任务的 restart 必须走独立重启流程，按原任务 URI 重新创建新任务，并且列表要直接显示 aria2 的 `errorMessage`，不要只给泛化报错。
- 症状：同名直链任务反复失败并报 `Detected a change in piece length`。原因：下载目录残留了旧的 `.aria2` 控制文件，里面记录的 piece length 与当前任务不一致。预防：用户提示里直接说明是续传控制文件冲突；失败任务执行“重新开始”时，先挪开冲突的 `.aria2` 控制文件，再重建任务。
- 症状：用户点击“移除”时预期未完成文件一起被清掉，但 aria2 默认只移除任务记录，磁盘残留仍在。原因：直接调用 `aria2.remove` / `aria2.removeDownloadResult` 不会处理本地文件。预防：未完成任务的移除必须走面板后端专门接口，直接删除对应文件和 `.aria2` 控制文件；已完成任务则只移除历史记录，避免误伤成品文件。
- 症状：面板设置里“刷新间隔”显示已保存，但任务轮询速度没有变化。原因：前端轮询定时器写死为常量，没有在初始化和设置保存后重新装配。预防：轮询间隔必须从 `/api/config` 读取，并在设置页保存成功后立即重建前端定时器。
- 症状：MCP 资源把内置 aria2 RPC 地址和托管端口直接暴露给外部客户端，和“只能通过面板代理访问”的边界冲突。原因：资源层复用了内部配置对象，未区分面板侧公开信息和内网运行参数。预防：MCP `resources/read` 只能暴露面板代理路径、面板 Secret 和受控配置，不要返回内置 aria2 的监听地址或托管端口。
- 症状：上传种子和提交链接使用同一套高级参数 UI，但只有链接创建真正带上了这些选项。原因：`upload-torrent` 接口只收文件，未透传目录、限速、Header 和 BT 参数。预防：创建任务的多条入口必须共用同一份 options 构造逻辑，保证种子和 URL 提交行为一致。
- 症状：all-in-one 的 Linux 内置 `aria2c` 下载 HTTPS 资源时提示 `SSL/TLS handshake failure: unable to get local issuer certificate`，但同机 `curl`/`openssl` 校验证书正常。原因：打包进去的 aria2 运行时不会稳定继承目标系统的 CA 查找路径，导致 OpenSSL 找不到系统证书包。预防：托管 aria2 启动时如果用户未显式配置 `ca-certificate`，自动探测常见系统 CA bundle 路径并追加 `--ca-certificate=<path>`。
- 症状：任务列表和详情里的失败原因直接显示 aria2 原始英文报错，用户难以快速判断问题。原因：`tellActive` / `tellWaiting` / `tellStopped` / `tellStatus` 返回的 `errorMessage` 被原样透传到前端。预防：在后端统一本地化 aria2 任务结果中的 `errorMessage`，把磁盘空间不足、证书链失败、权限不足、重复种子等常见错误翻成中文用户提示。
- 症状：任务详情想增加 AriaNg 风格的区块视图时，如果只依赖仪表盘已有字段，无法拿到实时的分片完成图。原因：区块展示依赖 aria2 `tellStatus` 的 `bitfield`，而列表轮询默认只拉基础状态字段。预防：区块页签按当前选中任务单独调用 `aria2.tellStatus` 拉取 `bitfield`、`numPieces`、`pieceLength` 等字段，再在前端按固定上限聚合成网格，避免对整页轮询造成额外负担。
- 症状：Peer 列表如果只显示地址和速率，或者只给一个连续百分比条，看不出对端真实持有哪些离散分片，和 AriaNg 的诊断体验差很多。原因：`aria2.getPeers` 返回里已经有 `bitfield`，但前端没有结合任务 `numPieces` 把它渲染成分片地图。预防：Peer 视图直接用 peer `bitfield` + 当前任务 `numPieces` 聚合成固定密度的小块地图，保留完成/部分完成/未完成三态，并跟随任务轮询刷新。
- 症状：区块页签如果按当前分片数量动态计算列数，任务之间的小块尺寸会忽大忽小，视觉对比很差。原因：网格列数跟着 `blockCells.length` 变化，导致单格宽度被容器二次拉伸。预防：区块页签的分片块使用固定尺寸网格，只让换行数变化，不让单块尺寸随任务变化。
- 症状：公开节点订阅这种内容型功能如果做成后端持久化配置，会把一组静态内置地址演变成配置迁移和接口维护负担。原因：需求本质只是展示若干公开源并便于复制，不涉及用户私有状态。预防：这类“内置资源导航”优先做成前端静态数据页，只在确实需要用户自定义或服务端同步时再引入后端配置。
- 症状：用户提到“订阅源”时，若不先确认是代理节点订阅还是 BT Tracker 订阅，很容易把实现方向带偏。原因：两者都叫订阅，但 aria2 只原生支持 `bt-tracker` 这类 Tracker 列表，并不直接消费代理节点订阅。预防：需求包含“订阅”“节点”“aria2 生效”时，先确认订阅对象；如果是 Tracker，直接对接 `bt-tracker`，不要引入代理内核方向。
- 症状：节点防护如果先写配置、后应用系统防火墙，一旦 `pfctl` 或 Linux 防火墙权限不足就会出现“配置里已封禁、系统里没拦住”的假成功。原因：节点级封禁最终依赖系统防火墙，而当前进程未必具备启用或刷新规则的权限。预防：节点防护的封禁必须以防火墙规则成功应用为准；应用失败时要回滚配置，并明确告诉用户节点尚未真正封禁。
- 症状：节点防护如果只适配单一防火墙，会让用户为了面板额外安装特定防火墙。原因：Linux 发行版上的防火墙能力并不统一，`firewalld`、`ufw`、`nftables`、`iptables` 的可用性依环境不同。预防：节点防护必须优先复用系统已有防火墙，按可用性依次探测并回退，不要要求用户为了面板单独安装特定防火墙。
- 症状：自动封禁如果没有评分机制，只要识别到可疑 Peer 就直接加入黑名单，容易误封正常节点。原因：可疑节点识别和自动封禁是两级决策，前者是信号，后者必须基于阈值。预防：节点防护的自动封禁必须以评分阈值为条件，只有达到阈值的节点才允许自动封禁，并在页面上明确展示当前阈值。
- 症状：节点防护页面在没有系统防火墙权限时仍然允许点击操作，会让用户误以为封禁命令已经生效。原因：后端虽然会返回失败，但前端没有把“无权限”提升为页面级锁定态。预防：权限检测失败时，页面要直接加遮罩禁止操作，并给出用户视角的修复提示，而不是只在按钮点击后被动报错。
- 症状：任务详情的区块页签在部分任务上会整片空白。原因：aria2 并不总会返回 `bitfield`，尤其在某些状态下只有 `numPieces`、`completedLength`、`totalLength`，前端如果只按 `bitfield` 解码就会把所有格子渲染成空。预防：区块视图在缺少 `bitfield` 时要回退到按完成度估算的固定网格，而不是直接显示空白。
- 症状：任务详情的 Peer 分片地图看起来像是空白，即使 Peer 行本身已经存在。原因：Peer 视图原先只在拿到 `peer.bitfield` 时才生成地图格子，样式上空格子也没有边框；一旦某个 Peer 没回传 `bitfield`，或者全是未完成格子，就会退成纯文字或视觉上像整块空白。预防：Peer 分片地图也要像区块页一样保留固定网格回退，并为空格子加可见边框，保证“无 bitfield”和“全未完成”都仍然有明确地图轮廓。
- 症状：登录态失效后，后台页面仍然原地显示“请先登录”，会让管理台看起来像是内部报错。原因：请求层把 401 当作普通业务错误继续抛给页面组件。预防：前端请求层遇到 401 时要立刻跳转回 `/login`，不要再让业务页面弹未登录提示。
- 症状：登录态失效后页面一直停在“AriaMX 正在启动”，没有真正回到登录页。原因：请求层在 401 时返回了一个永不结束的 Promise，导致启动流程和页面等待都被挂死。预防：401 触发登录页跳转时必须立即中断当前请求，不能用悬空 Promise 充当“跳转中”状态。
- 症状：`bt-tracker` 保存多个地址后重新打开只像是只剩一个，`aria2.conf` 里还会出现一行 `bt-tracker=` 后面跟多行裸 URL。原因：面板把多 tracker 文本按换行原样写进单个配置值，而 aria2 配置文件要求这类列表值写成单行逗号分隔。预防：编辑器可以按多行展示，但保存和写 `aria2.conf` 时必须把 `bt-tracker`、`bt-exclude-tracker`、`no-proxy` 规范化为逗号分隔单行值。
- 症状：节点订阅开启后，如果 aria2 全局选项页仍允许直接编辑 `bt-tracker`，会出现“订阅刚写入、手工又覆盖”或“手工改完又被订阅覆盖”的冲突。原因：节点订阅和 `bt-tracker` 编辑器操作的是同一项配置，但页面没有把这层所有权表达清楚。预防：节点订阅开启时，`bt-tracker` 必须在前端显示为只读，后端保存接口也要拒绝对该项的直接修改。
- 症状：开启节点订阅保存时，页面只显示“请求失败，请稍后重试。”，用户无法判断是拉取订阅源失败还是应用到 aria2 失败。原因：节点订阅保存链路先写配置再应用，且前端在接口返回非 JSON 或空错误体时只剩通用兜底文案。预防：节点订阅应先获取订阅源，再保存配置并应用；前端对 `/api/tracker-subscription` 要提供更明确的兜底错误提示。
- 症状：用户要求把默认 aria2 配置对齐到外部参考 `aria2.conf` 时，如果直接照搬 RPC、路径、脚本和监听端口，会和面板托管模式冲突，甚至导致启动失败或暴露不该开放的端口。原因：第三方 `aria2.conf` 面向裸 aria2 进程，而 AriaMX 会接管 RPC、安全和运行目录。预防：同步第三方默认配置时，只对齐通用下载/BT/性能项；RPC、安全、运行路径、脚本钩子和高冲突监听项继续由面板托管。
- 症状：任务详情里固定展示 `Server`、`Tracker`、BT 字段等信息时，直链任务或普通任务会出现空页签和无意义字段。原因：详情页按固定模板渲染，没有根据任务实际数据裁剪。预防：任务详情只展示当前任务真正有数据的页签和字段，例如没有 `Server`/`Tracker`/BT 信息时不要占位显示。
- 症状：all-in-one 模式下管理后台继续展示内置 aria2 的 RPC 端口，会让用户误以为需要手动维护该端口。原因：实际外部入口始终是面板 `/jsonrpc`，内置 aria2 端口只是面板内部托管细节，且会自动探测空闲端口。预防：管理后台不要展示或配置内置 RPC 端口，只保留面板代理入口信息。
- 症状：节点防护页在增加控制项后，封禁名单会把内容区继续往下撑，超出固定工作区高度。原因：页面布局只有顺序堆叠，没有给名单区域单独的滚动边界。预防：节点防护页的控制区放到顶部固定区域，风险列表和封禁名单放进受限高度的分栏滚动区，避免突破页面高度。
- 症状：用户需要外部 AriaNg 通过面板 `/jsonrpc` 走 WSS 接入时，WebSocket 会被固定的同源校验直接返回 403。原因：面板层 `CheckOrigin` 只允许 `Origin.Host == Host`，没有为第三方前端接入保留可配置策略。预防：`/jsonrpc` 的同源校验必须做成可持久化配置，至少支持关闭、同源、白名单三档，并在设置页同步暴露白名单编辑入口。
- 症状：设置页随着配置项增多会退化成单列长表单，关键信息和保存操作被埋没，条件字段展开后也容易破坏页面整洁。原因：设置页缺少信息层级和分组，只按字段顺序堆叠。预防：设置页优先采用“顶部概览 + 分组卡片 + 底部操作区”结构，条件字段在所属卡片内展开，避免撑乱整个工作区布局。
- 症状：设置页在白名单这类条件字段展开后，保存区会被覆盖，页面还会被拉长触发额外滚动。原因：根面板和中间内容区同时承担滚动，底部操作区没有独立占位。预防：设置页根容器保持固定网格，只让中间内容区滚动，底部操作区单独占位并带顶部边界。
- 症状：面板密码协议调整后，如果继续保留旧密码格式兼容和迁移分支，会让登录与改密协议长期双轨并存，增加判断复杂度。原因：把本应一次性切换的部署决策做成了运行时兼容逻辑。预防：若明确按首次部署处理，密码协议直接统一为“前端 SHA-256，后端对该值做 PBKDF2-SHA256 存储”，不要保留旧方案迁移分支。
- 症状：本地用 Caddy 给面板套 `tls internal` 时，服务看起来配置正确却起不来，或者起得来但浏览器/`curl` 仍报证书不受信任。原因：默认自动 HTTPS 会额外抢占 `:80` 做跳转，本机若已有服务占用就会启动失败；即便成功签发本地证书，Caddy 也不一定能自动把本地根证书装进当前用户信任链。预防：本地固定 HTTPS 端口反代时在全局配置加 `auto_https disable_redirects`，并在 macOS 上把 Caddy 本地 CA 的 `root.crt` 手动加入登录钥匙串后再验证。
- 症状：同一个 BT 任务在面板里的 Peer 分片地图看起来和 AriaNg 不一致，常见表现是整条几乎全黄或全蓝。原因：如果把大量 piece 过度聚合成少量格子，再把所有混合格统一渲染成单一 `partial` 样式，离散分布细节会被抹平。预防：Peer 分片地图要保持足够高的格子密度，并让 partial 格按真实完成占比显示填充，而不是固定半黄半蓝。
- 症状：本地重启 AriaMX 开发实例时直接报 `load config: open ariamx.json: permission denied`。原因：工作区根目录的 `ariamx.json` 被 root 以 `0600` 写入，当前用户无法读取，新的 `go run` 进程也就无法复用这份配置。预防：本地开发不要把主配置写成 root-only；若现场配置已不可读，先改用同目录下当前用户可写的新配置文件启动，避免卡住前端验证。
- 症状：用户要求区块页“和 AriaNg 一样”时，如果仍把 `bitfield` 聚合成固定数量格子，地图会变成近似概览图，和 AriaNg 的逐分片视图明显不一致。原因：AriaNg 的区块页是按 `pieceCount` 逐个 piece 渲染格子，不做固定 40/80/960 格这种预聚合。预防：当需求明确要对齐 AriaNg 区块地图时，有 `bitfield` 就按 `1 格 = 1 分片` 直接绘制；只有在 aria2 没返回 `bitfield` 时，才允许退回估算图，并明确提示这不是精确分片地图。
- 症状：Peer 地图如果把对端 `bitfield` 压缩成固定格数，会出现整条大面积同色块，和 AriaNg 的 Peer 分片分布差异很大。原因：固定 40/80 格会把离散 piece 分布折叠成聚合状态，混合区块只能显示成概括性的部分完成。预防：当用户要求 Peer 地图不要聚合或要对齐 AriaNg 时，优先按 `peer.bitfield + task.numPieces` 逐 piece 渲染；只有在 Peer 缺少 `bitfield` 时才保留占位轮廓，不要再做固定格数压缩。
- 症状：用户要求 Peer 视图“像 AriaNg 那样线条”时，如果继续沿用逐 piece 小方格，虽然精度对了，但视觉形态仍和 AriaNg 明显不同。原因：AriaNg 的 Peer 视图用的是 `piece-bar` 风格的细长条带，不是 `piece-map` 方格；两者都基于真实 bitfield，但展示控件不同。预防：区块页与 Peer 页要分开处理，区块页可用逐 piece 方格，Peer 页若要求对齐 AriaNg，应改成基于 bitfield 的连续线条条带，而不是继续堆小方格。
- 症状：Peer 地址后面需要显示 `(BitComet)` 这类客户端信息时，单靠 IP/端口无法补出来源客户端。原因：aria2 `getPeers` 返回的客户端线索主要在 `peerId`，需要按常见 BT client peerId 前缀做识别。预防：Peer 列展示客户端名时，优先从 `peerId` 解析常见前缀（如 `-UT`、`-qB`、`exbc` 等）；识别不出来就留空，不要伪造客户端名称。
- 症状：本地 Peer 列明明有 `peerId`，界面却完全看不到 `(qBittorrent)`、`(Deluge)` 这类客户端标记。原因：aria2 `getPeers` 返回的 `peerId` 是 URL 编码字符串，例如 `%2DqB4520...`，如果前端直接拿原串做 `startsWith('-qB')` 之类判断会全部失效。预防：Peer 客户端识别前先做 URL 解码，再按 BEP20/Azureus 风格前缀表识别客户端；tooltip 若展示 `peerId`，也应展示解码后的值。
- 症状：BT 任务详情里的 Peer 页如果只展示远端 peers，没有单独的本机行，那么无 Peer 或连接很少时状态页信息会残缺，甚至页签直接消失。原因：远端 `aria2.getPeers` 结果只代表对端连接，不代表本机当前下载/做种状态；把页签显示条件绑死在远端 peers 数量上会漏掉本机状态。预防：BT 任务的该页应按“状态”理解，至少始终展示一行本机状态（本机进度条、上下行速度、做种状态），远端 peers 再附加显示；页签显示条件也应允许 BT 任务在无远端 Peer 时仍可见。
- 症状：侧栏左上角品牌区如果直接堆“面板版本 / aria2 版本”等运行时信息，会让导航头过于像诊断面板，识别性差且视觉噪声大。原因：品牌区承担的是产品识别和导航起点，不适合塞高频无操作价值的技术元数据。预防：左上角品牌区只保留稳定的面板标识与简短产品文案；版本号、Secret、RPC 地址等运行时信息统一留在“连接信息”页查看。
- 症状：整站皮肤如果只在前端写死图片地址，用户一旦更换接口或设备，登录页和主面板视觉就会恢复默认且无法复用。原因：皮肤图来源属于面板配置，不是单次会话状态。预防：整站皮肤至少落到 `panel.skinEnabled`、`panel.skinName`、`panel.skinApiTemplate` 三个持久化字段，并约定模板用 `{skin}` 占位解析真实图片地址。
- 症状：某些随机图片 API 在浏览器里看似可访问，但作为 HTTPS 页面背景图时却完全不生效。原因：这类接口可能先重定向、返回不稳定响应头，甚至从 HTTPS 再跳到 HTTP；一旦改成浏览器直连加载，混合内容、鉴权和最终响应类型都会直接决定背景能否显示。预防：背景图片功能要让浏览器自己预加载并直接使用最终 URL，设置说明里明确要求填写浏览器可直接访问、最终仍返回真实图片内容的 HTTPS 地址，不再通过服务端代理兜底。
- 症状：用户明明已经开启整站皮肤，但登录页完全看不到，登录后也容易误以为“没生效”。原因：皮肤样式如果只在登录后的私有配置链路里加载，未登录页就拿不到；同时主界面的深色渐变和面板底色过重，会把背景图压成只有轻微氛围。预防：登录页也要走公开的面板样式配置读取，且开启皮肤时要显著降低整站蒙层和卡片底色，让背景图在登录前后都能被肉眼明确感知。
- 症状：用户想把“整站皮肤”仅当成背景图功能使用时，如果设置页继续暴露“皮肤名”和单独概览卡片，会让功能概念显得过重，还会和“只是换背景图”这件事脱节。原因：底层实现仍保留了 `skinName` 这类模板参数，但当前用户心智和页面用途都已经收敛成单一背景图配置。预防：设置页文案统一用“背景图片”，去掉背景图概览卡片和皮肤名输入，只保留开关与图片地址，避免把实现细节直接暴露给用户。
- 症状：整站背景图开启后，某些 section 的角上会同时看到圆角边缘和一层直角底色，像是“圆角里套了个方角”。原因：半透明背景叠加 `backdrop-filter` 时，如果没有同时收紧背景裁切和层隔离，模糊层或子层底色会在圆角边缘产生直角重影。预防：这类透明面板统一补 `background-clip: padding-box`、`isolation: isolate` 和 `-webkit-backdrop-filter`，并优先去掉外层大面板的重阴影，避免圆角边缘出现直角残影。
- 症状：背景图模式下，有些面板角部只有在保留 `box-shadow` 时才会出现“圆角旁边带直角”的错觉，关掉阴影后问题就消失。原因：透明背景、毛玻璃和阴影叠在一起时，阴影边界会把圆角外缘又描出一层近似方角的轮廓，尤其在深色半透明卡片上更明显。预防：背景图模式下这类 panel/card 优先去掉或极限减弱 `box-shadow`，改靠轻边框和透明层次表达层级，不要继续依赖厚阴影。
- 症状：背景图模式下卡片已经透明，但输入框、下拉框、多行输入仍是实心底，视觉上像表单没有跟随皮肤。原因：背景图样式只覆盖了 panel/card，未同步覆盖 `input`、`textarea`、`select` 的默认态、聚焦态和禁用态。预防：开启背景图时，表单控件也要统一切到半透明背景、轻边框和毛玻璃效果，否则整套界面会割裂。
- 症状：背景图模式下仍会残留少数“实心卡片”，尤其是组件内 scoped 样式定义的卡片，以及空状态这类不走常规 panel/card 体系的块。原因：只改全局 `.panel`、`.settings-section-card` 之类选择器不够，局部组件样式和 `empty-state` 这类独立块不会自动继承。预防：做整站皮肤时要额外排查 scoped 组件卡片、空状态、弹窗、表格行和值展示块，必要时改成 CSS 变量承接全局皮肤值。
- 症状：浏览器标签页 favicon 看起来不对或长期不刷新，即使资源文件已更新。原因：浏览器常优先命中缓存的 `.ico`，而且如果先声明 ico、再声明 svg，现代浏览器也可能继续沿用旧的 ico。预防：`index.html` 里优先声明 `svg` favicon，再提供 `shortcut icon` 的 `.ico` 回退，并同步更新版本参数；如果需要 ico，重新按 16/32/48/64 多尺寸生成。
- 症状：Aria2 配置页的选项行会被裁切，说明文本、类型标签和输入框下半部分显示异常。原因：为了修背景图模式的圆角与阴影问题，把 `option-row` 也套用了 `overflow: hidden` 和卡片裁切逻辑，但这个列表本身包含大量多行说明，行高必须由内容自然撑开。预防：`Aria2OptionsPanel` 的 `option-row` 不能按普通卡片处理，不要给它加裁切或强制卡片化效果，优先保留自然高度和顶部对齐布局。
- 症状：背景图开启后，右侧内容区滚动条会显得很突兀，和整套半透明界面不协调。原因：项目默认没有给这些滚动容器定制 scrollbar，浏览器会直接使用系统默认样式，轨道和滑块对比过强。预防：对右侧主要滚动容器统一提供细滚动条样式，并在背景图模式下通过 CSS 变量进一步弱化 thumb/track 对比度。
- 症状：任务详情的“连接状态”里 peer 行下载/上传方向看起来和 AriaNg 相反，用户会觉得“我从对方下载，为什么对方这一行还显示下载”。原因：aria2 `getPeers` 原始速度字段是按本机视角给出的，而 AriaNg 在展示 peer 列表时会先交换 `downloadSpeed` 与 `uploadSpeed`，改成按对端 peer 视角显示。预防：如果要和 AriaNg 对齐，远端 peer 行必须交换这两个字段；本机补充行则继续保留本机视角。
