# AI 项目经验

- 症状：生产构建后的页面出现模板自带外部跳转入口，浏览器验证时会离开本地面板。原因：Vite 模板默认启用了 `vite-plugin-trae-solo-badge`。预防：面向实际发布的 Web 面板必须移除模板广告/徽标插件，并检查 `index.html` 标题与语言。
- 症状：本地冒烟测试访问 `127.0.0.1:18080` 命中了其他进程。原因：已有 IPv4 监听进程占用同端口，而测试服务监听在 IPv6 通配地址。预防：冒烟测试使用明确的 `127.0.0.1:<port>` 绑定，并用未占用端口。
- 症状：`.gitignore` 中写入 `ariamx` 会把源码目录 `cmd/ariamx/` 一并忽略。原因：无斜杠的 gitignore 规则会匹配任意层级同名文件或目录。预防：忽略根目录二进制时使用 `/ariamx`。
- 症状：冒烟验证中设置 `ARIAMX_ADMIN_PASSWORD=admin` 后仍无法登录。原因：`ARIAMX_ADMIN_PASSWORD` 只在配置文件首次创建时生效，已存在配置不会覆盖密码。预防：认证相关冒烟测试使用全新的 `ARIAMX_CONFIG` 路径。
- 症状：Vite 开发环境打开页面后 `/api` 请求会打到前端端口。原因：开发模式没有把 `/api` 转发给 Go 后端。预防：`vite.config.ts` 中配置 `/api` 代理到本地 Go 服务端口，并在 README 记录 dev 启动顺序。
- 症状：开发模式下前端能访问后端 API，但直接请求 `/mcp` 会返回前端 404。原因：Vite 代理只配置了 `/api` 和 `/jsonrpc`，没有把新增的 MCP 入口一起转发。预防：凡是前端开发态要直连的后端入口，都要同步写进 `vite.config.ts` 代理，MCP 场景至少包含 `/api`、`/jsonrpc` 和 `/mcp`。
- 症状：用户希望关闭 MCP 时不仅隐藏连接信息，还要让 `/mcp` 真正失效。原因：只改前端展示会留下后端入口仍可调用的假开关。预防：MCP 开关必须落到 `panel.mcpEnabled` 持久化配置，并同时控制设置页、连接信息页和后端 `handleMCP` 的实际访问行为。
- 症状：aria2 设置页只有裸 `key=value`，用户难以对应 AriaNg 的设置入口。原因：只读取了 `aria2.getGlobalOption`，没有维护 AriaNg 的分类和中文文案元数据。预防：全局选项页保留 AriaNg 的基本、HTTP/FTP/SFTP、HTTP、FTP/SFTP、BitTorrent、Metalink、RPC、高级分组，并为常用项提供中文名称、说明、只读和枚举信息。
- 症状：aria2 设置页新增高级 `key=value` 编辑区后 UI 被撑乱。原因：raw textarea 常驻在设置页下方，且 `.options-panel textarea` 全局选择器会影响该面板内所有 textarea。预防：高级编辑区默认折叠，并用 `.raw-option-editor textarea` 精确限定大文本框样式。
- 症状：aria2 设置页右侧标题在两栏结构外，右侧内容上下边界无法和左侧分类栏对齐。原因：标题和操作按钮不在 `.settings-layout` 同一网格内。预防：设置页标题、搜索和操作按钮都放进右侧 `.settings-content`，两栏容器只负责左右对齐。
- 症状：用户要求界面“像 AriaNg”，但局部组件调整后整体观感仍停留在玻璃拟态。原因：主题色、边框、按钮、列表和侧栏基线样式都集中在 `src/style.css`，只改单个组件无法完成整体迁移。预防：涉及整站风格切换时，优先统一全局设计 token 与共享容器/控件样式，再处理页面细节。
- 症状：用户提供了设计导出目录后，若只参考单个 HTML 页面，容易漏掉统一的颜色、字号和间距规范。原因：结构示例、设计 token 和说明分别散落在 `ariang-config-design.html`、`ariang-admin-style.css`、`ariang-admin-style-guide.md`。预防：遇到该目录下的样式对齐任务时，优先以 CSS 和 style guide 作为全局视觉基线，再用 HTML 对齐局部结构。
- 症状：用户要求“经典皮肤 / 设计稿皮肤”按账号保留时，只靠 `localStorage` 会在清缓存或换设备后丢失。原因：皮肤状态属于面板配置，不是纯浏览器临时偏好。预防：皮肤字段写入 `panel.theme` 并通过 `/api/config` 下发；前端启动和登录成功后都要重新应用服务端主题。
- 症状：all-in-one 模式启动内置 aria2 时，若运行时目录用相对路径或未预创建 `session.dat`，`aria2c` 会直接启动失败。原因：进程执行路径依赖当前工作目录，且 `--input-file` 指向的 session 文件必须存在。预防：内置 aria2 的状态目录统一转成绝对路径，并在首次启动前创建空的 `session.dat`。
- 症状：all-in-one 发布包体积异常膨胀到约 93M。原因：`//go:embed runtime/*` 会把 `darwin-arm64`、`linux-amd64`、`linux-arm64` 三套 aria2 运行时归档全部打进每一个二进制。预防：all-in-one 构建必须使用按 `GOOS/GOARCH` 拆分的 build tag embed，并让 CI 按平台分别产出发布包。
- 症状：面板里修改 `rpc-listen-port` 后界面显示已保存，但内置 aria2 实际仍监听旧端口。原因：Aria2 选项里的 `rpc-listen-port` 与面板托管字段 `ManagedRPCPort` 是两套状态，保存时没有同步。预防：all-in-one 模式下凡是由面板启动参数接管的 aria2 选项，保存时必须先映射回托管配置，再决定是否重启 aria2。
- 症状：修改内置 aria2 的 RPC 监听端口后，旧 aria2 先被停掉，但新 aria2 因端口已占用启动失败，结果面板还在、aria2 却掉线。原因：重启前没有先探测新端口是否可绑定，本机残留进程或其他服务占用端口时会把切换过程做成半失败。预防：保存 `rpc-listen-port` 前必须先探测 `127.0.0.1`/`::1` 端口可用性；若不可用，直接拒绝保存并保留当前 aria2 进程。
- 症状：移除旧皮肤枚举后，已存在配置仍保存 `design`/`classic`，前端和后端会出现主题状态不一致。原因：主题字段属于持久化配置，删除旧样式时如果不做迁移，历史配置不会自动收敛到新枚举。预防：主题重构时保留 `panel.theme` 兼容迁移逻辑，并把显示模式拆到独立的 `panel.colorMode` 字段；`design -> light`，`classic -> dark`。
- 症状：面板异常退出后，残留的内置 aria2 仍占着原 RPC 端口，下一次启动面板会误判成“端口冲突”而无法启动。原因：之前启动逻辑只检查端口是否可绑定，没有校验占用者是否就是上一轮遗留的同 secret aria2 进程。预防：all-in-one 启动时若发现 RPC 端口已占用，先用当前 `rpcUrl + rpcSecret` 调 `aria2.getVersion`；如果鉴权成功，就直接复用旧 aria2，而不是重复拉起。
- 症状：AIO 场景里，内置 RPC 固定端口被其他进程占用时，面板无法继续托管 aria2。原因：单一固定端口在用户本机容易与已有服务冲突。预防：当目标端口不可用且也不是可复用的旧 aria2 时，从目标端口开始按 `+10` 自动步进寻找空闲端口，并把最终端口写回配置；同时在主进程收到 `SIGINT`/`SIGTERM` 时主动停止内置 aria2，减少残留进程。
- 症状：主题模式加入“跟随系统”后，如果只把 `system` 原样写进 DOM 的 `data-mode`，现有深浅色 CSS 不会生效，也不会随系统切换自动更新。原因：样式层实际只识别解析后的 `light`/`dark`。预防：配置层允许保存 `panel.colorMode=system`，但前端应用主题时必须先解析成实际模式写入 DOM，并监听 `prefers-color-scheme` 变化实时重算。
- 症状：本地明明已经改了代码，但浏览器里看到的还是旧界面。原因：旧的 `dist/ariamx` 二进制实例仍在运行，而不是当前工作区代码启动的开发态服务。预防：当用户要求切到最新版本或实时查看改动时，先停掉旧的发布二进制实例，再用当前工作区代码启动后端，并确认前端访问的是对应的开发服务。
- 症状：前端源码里已经有新选项，但通过 Go 服务访问页面时仍看不到。原因：Go 端提供的是编译时嵌入的 `internal/web/dist`，仅重启 `go run ./cmd/ariamx` 不会自动把 `src/` 变更编进页面。预防：只要修改了前端可见行为且用户通过后端端口访问页面，就先执行 `pnpm run build` 更新 `internal/web/dist`，再重启 Go 服务。
- 症状：外部程序接入 `/jsonrpc` 时，如果直接复用 aria2 内部 secret，会把面板代理层和内置 aria2 的信任边界混在一起。原因：面板作为 RPC 中间件需要有独立鉴权，而 aria2 内部 secret 只应存在于面板到 aria2 的内部链路。预防：外部调用统一校验 `panel.rpcSecret`，并保证它永远不与 `aria2.rpcSecret` 相同；AIO 模式下内置 aria2 继续只监听 `127.0.0.1`。
- 症状：连接信息页需要展示面板 RPC Secret 时，若直接开放到匿名接口会扩大敏感信息暴露面。原因：secret 属于高敏感配置，只适合在受控页面里最小范围展示。预防：默认不要在匿名接口泄露 secret；只有在用户明确要求时，才通过已登录接口返回并显示在受保护页面中。
- 症状：改成面板独立 RPC Secret 后，AriaNg 默认的 `token:<secret>` 参数鉴权会失效。原因：AriaNg 不会主动补 `Authorization: Bearer`，而是把 secret 放进 JSON-RPC `params[0]`。预防：面板 `/jsonrpc` 需要兼容校验外部传入的 `token:<panel.rpcSecret>`，但校验通过后仍要丢弃它，并替换成内部 `aria2.rpcSecret` 转发，不能把外部 token 原样透传给 aria2。
- 症状：给面板增加 MCP 能力后，如果另起一套 secret 或匿名开放工具列表，外部客户端配置和信任边界会变复杂。原因：MCP 和 `/jsonrpc` 本质上都是面板代理层入口。预防：`/mcp` 继续复用 `panel.rpcSecret`，并在登录页内直接展示可复制的 MCP 地址和客户端配置示例。
- 症状：MCP 只实现 `tools/list` / `tools/call` 时，客户端虽然能调 aria2 工具，但无法按标准发现资源、Prompt，也不能对任务 GID 或资源模板参数做补全。原因：最小版 MCP 只覆盖了 tools primitive，没有把 `resources`、`prompts` 和 `completion` 一起实现。预防：AriaMX 的 MCP 服务保持在同一协议版本下完整声明并实现 `tools + resources + prompts + completion`，资源 URI、Prompt 名称和补全参数要和面板现有功能保持一致。
- 症状：失败任务点击“重新开始”后直接从列表消失，用户误以为面板把任务吞掉了。原因：按钮实际绑定的是 `aria2.removeDownloadResult`，删除了历史结果而不是重新创建下载。预防：失败任务的 restart 必须走独立重启流程，按原任务 URI 重新创建新任务，并且列表要直接显示 aria2 的 `errorMessage`，不要只给泛化报错。
- 症状：同名直链任务反复失败并报 `Detected a change in piece length`。原因：下载目录残留了旧的 `.aria2` 控制文件，里面记录的 piece length 与当前任务不一致。预防：用户提示里直接说明是续传控制文件冲突；失败任务执行“重新开始”时，先挪开冲突的 `.aria2` 控制文件，再重建任务。
- 症状：用户点击“移除”时预期未完成文件一起被清掉，但 aria2 默认只移除任务记录，磁盘残留仍在。原因：直接调用 `aria2.remove` / `aria2.removeDownloadResult` 不会处理本地文件。预防：未完成任务的移除必须走面板后端专门接口，直接删除对应文件和 `.aria2` 控制文件；已完成任务则只移除历史记录，避免误伤成品文件。
- 症状：面板设置里“刷新间隔”显示已保存，但任务轮询速度没有变化。原因：前端轮询定时器写死为常量，没有在初始化和设置保存后重新装配。预防：轮询间隔必须从 `/api/config` 读取，并在设置页保存成功后立即重建前端定时器。
- 症状：MCP 资源把内置 aria2 RPC 地址和托管端口直接暴露给外部客户端，和“只能通过面板代理访问”的边界冲突。原因：资源层复用了内部配置对象，未区分面板侧公开信息和内网运行参数。预防：MCP `resources/read` 只能暴露面板代理路径、面板 Secret 和受控配置，不要返回内置 aria2 的监听地址或托管端口。
- 症状：上传种子和提交链接使用同一套高级参数 UI，但只有链接创建真正带上了这些选项。原因：`upload-torrent` 接口只收文件，未透传目录、限速、Header 和 BT 参数。预防：创建任务的多条入口必须共用同一份 options 构造逻辑，保证种子和 URL 提交行为一致。
- 症状：节点防护如果先写配置、后应用系统防火墙，一旦 `pfctl` 或 Linux 防火墙权限不足就会出现“配置里已封禁、系统里没拦住”的假成功。原因：节点级封禁最终依赖系统防火墙，而当前进程未必具备启用或刷新规则的权限。预防：节点防护的封禁必须以防火墙规则成功应用为准；应用失败时要回滚配置，并明确告诉用户节点尚未真正封禁。
- 症状：节点防护如果只适配单一防火墙，会让用户为了面板额外安装特定防火墙。原因：Linux 发行版上的防火墙能力并不统一，`firewalld`、`ufw`、`nftables`、`iptables` 的可用性依环境不同。预防：节点防护必须优先复用系统已有防火墙，按可用性依次探测并回退，不要要求用户为了面板单独安装特定防火墙。
- 症状：自动封禁如果没有评分机制，只要识别到可疑 Peer 就直接加入黑名单，容易误封正常节点。原因：可疑节点识别和自动封禁是两级决策，前者是信号，后者必须基于阈值。预防：节点防护的自动封禁必须以评分阈值为条件，只有达到阈值的节点才允许自动封禁，并在页面上明确展示当前阈值。
- 症状：节点防护页面在没有系统防火墙权限时仍然允许点击操作，会让用户误以为封禁命令已经生效。原因：后端虽然会返回失败，但前端没有把“无权限”提升为页面级锁定态。预防：权限检测失败时，页面要直接加遮罩禁止操作，并给出用户视角的修复提示，而不是只在按钮点击后被动报错。
- 症状：登录态失效后，后台页面仍然原地显示“请先登录”，会让管理台看起来像是内部报错。原因：请求层把 401 当作普通业务错误继续抛给页面组件。预防：前端请求层遇到 401 时要立刻跳转回 `/login`，不要再让业务页面弹未登录提示。
- 症状：登录态失效后页面一直停在“AriaMX 正在启动”，没有真正回到登录页。原因：请求层在 401 时返回了一个永不结束的 Promise，导致启动流程和页面等待都被挂死。预防：401 触发登录页跳转时必须立即中断当前请求，不能用悬空 Promise 充当“跳转中”状态。
- 症状：`bt-tracker` 保存多个地址后重新打开只像是只剩一个，`aria2.conf` 里还会出现一行 `bt-tracker=` 后面跟多行裸 URL。原因：面板把多 tracker 文本按换行原样写进单个配置值，而 aria2 配置文件要求这类列表值写成单行逗号分隔。预防：编辑器可以按多行展示，但保存和写 `aria2.conf` 时必须把 `bt-tracker`、`bt-exclude-tracker`、`no-proxy` 规范化为逗号分隔单行值。
- 症状：任务详情里固定展示 `Server`、`Tracker`、BT 字段等信息时，直链任务或普通任务会出现空页签和无意义字段。原因：详情页按固定模板渲染，没有根据任务实际数据裁剪。预防：任务详情只展示当前任务真正有数据的页签和字段，例如没有 `Server`/`Tracker`/BT 信息时不要占位显示。
- 症状：节点防护页在增加控制项后，封禁名单会把内容区继续往下撑，超出固定工作区高度。原因：页面布局只有顺序堆叠，没有给名单区域单独的滚动边界。预防：节点防护页的控制区放到顶部固定区域，风险列表和封禁名单放进受限高度的分栏滚动区，避免突破页面高度。
